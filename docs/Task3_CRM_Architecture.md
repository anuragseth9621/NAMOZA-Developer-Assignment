# Task 03 – CRM Integration Architecture

I would implement this as a custom backend integration using REST APIs rather than a no-code platform. Since the landing page is already a custom HTML/JavaScript application, a backend API provides better control over validation, error handling, logging, security, and future scalability than tools such as Zapier or Make.

The flow begins when the user submits the consultation form from the landing page. The frontend sends the patient's name, phone number, and clinic preference to a secure backend endpoint over HTTPS. Before writing anything to HubSpot, the backend first checks whether a contact with the same phone number already exists. Because HubSpot does not automatically deduplicate contacts by phone number, the backend calls the HubSpot CRM Contacts Search API and searches the phone property. If a matching contact is found, the backend updates that existing record with the latest information. If no match is returned, it creates a new contact using the HubSpot CRM Contacts API. The backend also sets **Source = "Google Ads - Consultation Landing Page"** and **Lead Status = "New Enquiry"** so that new enquiries are correctly categorized inside HubSpot. This prevents duplicate patient records while keeping HubSpot as the single source of truth.

Once the HubSpot operation succeeds, the backend immediately sends a confirmation message through the Karix WhatsApp Business REST API using the approved WhatsApp template. In parallel, it records the Google Ads conversion named **consultation_form_submitted** after the HubSpot operation succeeds by invoking the Google Ads Conversion API from the backend. Recording the conversion only after the CRM operation completes ensures that advertising data reflects successfully processed consultation requests instead of incomplete submissions.

The biggest operational risk is temporary unavailability of the Karix WhatsApp API. If that service is slow or unavailable, the two-minute delivery requirement could be missed even though the consultation has already been recorded in HubSpot. To handle this, the backend places failed WhatsApp requests into a background retry queue with exponential backoff. This allows the user submission to complete successfully while the messaging service continues retrying automatically until delivery succeeds or the retry policy is exhausted.

To meet the two-minute SLA, the backend records timestamps for message creation, API request, and successful Karix response. These metrics are published to CloudWatch, where API latency and queue age are monitored. A CloudWatch alarm is configured to notify the engineering team if any message remains in the retry queue for more than two minutes or if repeated API failures occur, allowing rapid investigation before the delay affects a larger number of patients.
